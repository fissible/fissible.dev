Two things change in a Station deployment: **code** and **content**. They ship on different cadences, answer to different approvers, and fail in different ways. Station treats them as two separate lanes.

> *Git remains the source of code provenance, not the enforcement mechanism for content package import.*

## The ownership boundary

### CI/CD owns

- pulling code
- installing dependencies
- building assets
- running migrations
- restarting services
- **content type PHP artifacts** (generated by the admin UI, shipped as code)

### Station owns

- per-tenant production editing policy
- content package creation (`station:content:export`)
- content package validation + import (`station:content:import`)
- content promotion auditability

Station does not ship code, does not run migrations, and does not create content types at import time. If a package references a content type that is not present on the target, the import fails closed and the operator is told to deploy code first.

## Why two lanes?

- **Different cadences.** Code goes out on pull-request merges; content goes out when editorial says "ship." They should not block each other.
- **Different reviewers.** Developers review code diffs in GitHub. Editors review content changes in Station. Conflating them trains each group to rubber-stamp the other's work.
- **Different failure shapes.** A bad code deploy can be rolled back by redeploying. Content rollback is different — "undoing" an entry means promoting the previous version, and the history needs to be inspectable in the CMS, not in git.
- **Different privacy shape.** Content can include PII, drafts, or embargoed material. It does not belong in a public git repository by default.

## The recommended workflow

### Code lane

1. Make code changes locally.
2. Commit.
3. Push a branch or tag.
4. CI deploys code and runs migrations.

### Content lane

1. Make content changes in local/dev or staging.
2. Review and approve them there.
3. Create a content deployment package:
   ```
   php artisan station:content:export --tenant=acme --out=releases/2026-04-18.zip
   ```
4. Commit the package file if git-tracked promotion is desired.
5. Push.
6. Import on the target environment (automatically via CI, or manually with approval):
   ```
   php artisan station:content:import releases/2026-04-18.zip --yes
   ```

### Operational note

Teams are free to use live production editing where it's appropriate. Controlled promotion is a supported workflow, not a mandate. See [Controlled vs. Live Editing](/station/controlled-vs-live-editing/) for when to choose which mode.

## What lives in a package

A package is a single `.zip` artifact containing:

- `manifest.json` — provenance + index, with a required `package_checksum` verified at import
- `entries/<content-type>/<slug>.json` — one file per included entry, captured as its latest published commit
- `media/<content-hash>.<ext>` — binary assets referenced by included entries

A package does **not** carry:

- content type definitions — those are code
- users, roles, or permissions
- form submissions
- site settings or module configuration
- tenant records — the target tenant must already exist

See the [Content Package CLI reference](/station/content-package-cli/) for all options, and [Controlled vs. Live Editing](/station/controlled-vs-live-editing/) for the per-tenant / per-content-type editing policy.
